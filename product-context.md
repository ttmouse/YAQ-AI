# 应擎 · 安全监管工作台 — 产品上下文文档

> 版本迭代：v1（初始框架）→ v2（三层模型重构）→ v3（AI 感知 + 持续跟踪闭环）→ v4（目标驱动 + 模块编排）→ v5（日常态势管理）
> 最后更新：2025-07-21
> 本文件记录产品方向讨论的关键决策、框架定义、设计原则和原型上下文，作为后续开发的一手参考。

---

## 一、产品定位

### 1.1 核心主张

> **把复杂的 SaaS 系统基于角色、基于场景，变成"事找人、数据找人"。**

工作台不是功能入口集合，而是**基于当前角色和场景，主动推送需要关注的事和需要看到的数据。**

### 1.2 领域确认

| 方向 | 当前选择 | 过去的假设（已排除） |
|---|---|---|
| **业务领域** | 日常安全监管 | 突发事件应急指挥 |
| **核心用户** | 安全监管管理者（监管局/应急办） | 值班指挥长 |
| **核心场景** | 日常监管简报、隐患跟踪、履职监督 | 事件处置全流程 |
| **节奏** | 每日/每周的常态化工作 | 事件驱动的紧急响应 |

> ⚠ **关键决策**：这两个领域的产品逻辑完全不同。"突发事件指挥"关注单事件的处置闭环；"日常监管"关注多主体的履职状态和趋势异常。当前产品方向是后者。v1 原型按应急指挥设计（事件列表 + 资源面板），v2 起全面转为安全监管工作台。

### 1.3 核心用户角色（当前假设）

- **安全监管管理者** — 关注辖区安全态势、各主体履职情况、异常问题的发现与督办
- 管理者在系统中的典型动作是：**看诊断 → 复制提示 → 发微信群/工作群 → 线下推进**
- 管理者的下级已经能看到系统内的数据，不需要上级在系统内再"触发"他们
- （后续可扩展：局领导、科室负责人、一线执法人员、企业安全管理员）

### 1.4 概念与文案原则

- **"事找人"、"数据找人"** 是设计理念，不作为页面显性文案展示
- 用户应该**感受**到系统主动推送了重要的事，而不是被告诉"这是事找人功能"
- 所有设计决策和布局逻辑继续由这两个理念驱动，但不出现在用户可见的 UI 中

## 二、三层智能模型 → 四层闭环模型（核心框架）

这是产品的核心交互模型，每一层代表工作台的一个能力层级。经过用户讨论，从三层扩展为四层（加入第 0 层和第 4 层）。

```
                    ┌─────────────────────────────────────┐
    第 0 层         │  预期基线（判断"正常"的前提）         │
                    │  计划值 / 历史均值 / 同类对比 / 阈值   │
                    └──────────┬──────────────────────────┘
                               │
                    ┌──────────▼──────────────────────────┐
    第一层         │  数据反馈 — 发生了什么                │
                    │  指标卡 + 基线对照 → "X vs 预期"     │
                    └──────────┬──────────────────────────┘
                               │
                    ┌──────────▼──────────────────────────┐
    第二层         │  问题诊断 — 哪里不正常  ← AI 驱动     │
                    │  三维交叉定位 + 根因分析 + 严重等级    │
                    │  关联分析 / 交叉验证 / 趋势推演      │
                    └──────────┬──────────────────────────┘
                               │  (一一对应)
                    ┌──────────▼──────────────────────────┐
    第三层         │  工作提示 — 下一步怎么做              │
                    │  可复制的管理文案 → 发微信群/工作群   │
                    │  (非系统内待办，是管理者输出工具)      │
                    └──────────┬──────────────────────────┘
                               │
                    ┌──────────▼──────────────────────────┐
    第四层         │  持续跟踪 — 闭环了吗  ← AI 持续更新   │
                    │  异常一经诊断即创建跟踪任务           │
                    │  每日更新进展，直到确认关闭            │
                    │  跟踪天数 + 进度条 + 最新进展         │
                    └─────────────────────────────────────┘
```

### 2.1 第 0 层 — 预期基线

> **为什么需要这一层？**
> "异常"是一个对比概念。没有基线，就无法判断"正常还是不正常"。

**基线类型：**

| 类型 | 示例 | 适用场景 |
|---|---|---|
| **计划对标** | 今日应完成 50 项检查，实际 42 项 | 任务完成率 |
| **历史对照** | 相比上周，隐患整改率下降 12% | 趋势变化 |
| **同类对比** | 张毅条线进度落后其他条线均值 17% | 横向比较 |
| **阈值告警** | 重大隐患超过 3 天未处置 | 超时规则 |

**产品含义：** 每个指标卡必须同时展示"当期值"和"基线对照"，用户看到的第一眼就能判断"好不好"。

### 2.2 第一层 — 数据反馈（程序化数据，无 AI）

**目标：** 告诉管理者发生了什么。

**核心指标（当前原型 5 个）：**

| 指标 | 值 | 基线 | 趋势 |
|---|---|---|---|
| 检查任务 | 已检查 368 家 | 计划 450 家 | 完成率 82% |
| 隐患新增 | 23 处 | 昨日新增 18 | ↑ 28% |
| 重大隐患 | 2 处 | 超 48h 未处置 | 需关注 |
| 整改超期 | 7 项 | 上周同期 4 项 | ↑ 75% |
| 关闭率 | 91% | 月度目标 95% | 差 4pp |

**产品原则：**
- 每个指标附带基线对照（vs 计划 / vs 昨日 / vs 上周）
- 异常指标自动标红/标橙，视觉上"跳出来"
- 指标数量控制在 5±2 个，避免信息过载
- 纯程序化处理，不标注 AI 身份

### 2.3 第二层 — 问题诊断（AI 驱动，核心差异化）

> **这不是"好看大屏"和"智能工作台"的分水岭。** 常规数据由程序处理，但基于数据定位问题、发现问题，这是 AI 的能力。

**目标：** 不只显示数据，而是指出哪里不正常。需要体现**分析深度**——关联分析、交叉验证、根因推断、风险推演。

**三维交叉定位框架：**

```
每一条异常 =
   谁异常（维度 A）× 什么事（维度 B）× 多严重（维度 C）
```

**维度 A — 按异常对象：**

| 子维度 | 示例 | 数据来源 |
|---|---|---|
| 按人/条线 | 张毅条线进度落后 17% | 人员履职数据 |
| 按片区 | 滨海片区重大风险覆盖不足 | 片区监管数据 |
| 按风险等级 | 重大风险主体欠检 40% | 风险分级数据 |
| 按企业 | 恒源化工连续 30 天未登录 | 企业行为数据 |
| 按主体类型 | 危化企业自查率仅 43% | 分类统计数据 |

**维度 B — 按异常类型：**

| 子维度 | 示例 | 对应数据 |
|---|---|---|
| 进度异常 | 检查任务完成率低于计划 | 今日检查任务完成率 |
| 覆盖异常 | 某风险等级/片区欠检 | 分片区监管 |
| 质量异常 | 隐患整改闭环慢、超期多 | 整改超期数量 |
| 行为异常 | 企业长期不登录、不自查 | 企业主体责任 |
| 趋势异常 | 某类隐患反弹、某片区上升 | 辖区安全形势 |

**维度 C — 按严重程度：**

| 等级 | 判定条件 | 展示 |
|---|---|---|
| 高优 | 涉及重大隐患/超期/高危企业 | 红色标签 |
| 中 | 明显偏离基线但未触发告警 | 橙色标签 |
| 低 | 轻微偏差，持续观察 | 灰色标签 |

**AI 分析深度（升级后的诊断卡片结构）：**

| 诊断项 | 事实层（程序处理的原始数据） | 分析层（AI 的推断） | 分析类型标签 |
|---|---|---|---|
| #1 张毅条线 | 完成率 48%（上周 56%） | 危化类企业自查率同步↓12%，3 家连续未登录同属该条线，根因在企业端配合度而非检查人力 | `关联分析` |
| #2 滨海片区 | 15 家仅检 6 家（覆盖率 40%） | 超 90 天未检 3 家与"未登录"企业名单完全重叠，系统性脱管风险 | `交叉验证` |
| #3 企业未登录 | 3 家超 30 天未登录 | 3 家在风险分级、规模、地域（滨海东侧）高度一致，区域性特征事件 | `特征分析` |
| #4 低风险抽查 | 抽查 18%，目标 30% | 当前力度下 19 家下月突破 180 天空窗阈值，长期零覆盖或致风险低估 | `风险推演` |

**AI 分析类型标签：**

| 标签 | 英文 key | 色系 | 含义 |
|---|---|---|---|
| `关联分析` | corr | 蓝色 | 发现不同维度数据之间的相关性 |
| `交叉验证` | xval | 红色 | 多个数据源交叉比对确认问题 |
| `特征分析` | trend | 绿色 | 识别异常对象的共同特征模式 |
| `风险推演` | proj | 蓝色 | 基于趋势预测未来风险 |

**产品原则：**
- 工作台"第一眼"应该是诊断结果，而不是数据仪表盘
- 每条异常包含：**事实层 + 分析层**，AI 的分析推理过程对用户可见
- 每条异常对应一条可复制的工作提示（诊断 → 行动的一一映射）

### 2.4 第三层 — 工作提示（AI 生成，管理者输出工具）

**目标：** 告诉管理者下一步怎么做。**不是系统内待办，而是管理者可复制外发的管理文案。**

> **关键定位变化：** 早期版本做成了"系统内待办"（勾选 + 进度跟踪 + 处置按钮）。但管理者的真实场景是：下级进系统已经能看到数据了，不需要上级在系统内再"触发"他们。管理者的真实动作是：**复制文案 → 发微信群/工作群 → 线下推进。**

**工作提示与异常诊断的一一对应：**

| 异常诊断 | 工作提示（可复制文案） |
|---|---|
| #1 张毅条线落后 · 企业端配合度下降 | @张毅 本周检查 25/52 家（完成率 48%），较上周↓8%。同步发现危化类企业自查率 43%（↓12%），建议从企业端配合度入手而非仅加大检查频次。 |
| #2 滨海片区覆盖不足 · 系统性脱管风险 | @滨海组 重大风险主体 15 家仅检 6 家（覆盖率 40%），超 90 天未检 3 家。交叉验证该 3 家处于连续未登录，存在系统性脱管风险，建议本片区内联合处置。 |
| #3 企业未登录 · 区域性特征事件 | 恒源化工、宏达建材、鑫盛机械连续 30 天未登录。3 家均为 B 级中小型工贸企业、集中于滨海片区东侧，建议纳入区域性专项排查。 |
| #4 低风险抽查不足 · 未来空窗风险 | 各条线：一般/低风险抽查 18%，低于目标 30%。当前力度下 19 家下月突破 180 天空窗阈值，建议启动批量补查。 |

**工作提示卡片结构（卡片内容不使用冗余文字，三段各司其职）：**

```
┌──────────────────────────────────────────────────────┐
│  01  │  标题（对象 + 问题类型，一句话精炼）            │
│      │  可复制文案（管理者直接可以发到群里的文字）      │
│      │  优先级标签 · 责任归属                         │
└──────────────────────────────────────────────────────┘
```

- **标题**：承担辨识功能（一眼知道这条说的是什么）
- **可复制文案**：承担输出功能（管理者直接复制使用）
- **元信息**：仅放补充数据（优先级、责任人）

**"复制全部"功能输出格式：**

```
【应擎 · 今日工作提示】
生成时间：2025-07-21 14:30:00

1. [第一条可复制文案]
2. [第二条可复制文案]
3. [第三条可复制文案]
4. [第四条可复制文案]

— 由应擎系统自动生成
```

### 2.5 第四层 — 持续跟踪（AI 持续更新）

**目标：** 异常一经诊断即创建持续跟踪任务，每日更新进展，直到确认关闭。

> **关键认识变化：** 早期版本设计为"昨日跟进"（一天一清）。但实际上问题的跟进可能不是一天两天的——要持续到确认关闭。

**跟踪状态机：**

```
         ┌──────────────────────────────────────┐
         │  异常诊断触发                          │
         └──────────┬───────────────────────────┘
                    │
         ┌──────────▼───────────────────────────┐
         │  创建跟踪任务（关联诊断 + 天数计数）    │
         └──────────┬───────────────────────────┘
                    │
         ┌──────────▼───────────────────────────┐
         │  每日更新进展                          │
         │  - AI 自动汇总                         │
         │  - 管理者手动补充                       │
         └──────────┬───────────────────────────┘
                    │
         ┌──────────▼───────────────────────────┐
         │  状态分类                              │
         │  ├─ ✅ 已解决                         │
         │  ├─ 🔄 进行中                        │
         │  ├─ ⏳ 未启动                        │
         │  └─ 🔴 升级（问题恶化或范围扩大）      │
         └──────────┬───────────────────────────┘
                    │
         ┌──────────▼───────────────────────────┐
         │  确认关闭？                            │
         │  是 → 移入已完成归档                    │
         │  否 → 继续跟踪                         │
         └──────────────────────────────────────┘
```

**跟踪任务卡片：**

```
┌───────────────────────────────────────────────┐
│  🔴 升级        已跟踪 15 天                   │
│  交通运输类企业自查率连续下降                   │
│  ← 异常诊断：趋势异常 · 自查率 43%（上月 55%）  │
│  连续 3 月恶化，已升级为专项督办方案。建议本周  │
│  专题会议讨论。                                │
│  ▓▓▓▓▓▓▓░░░  70% · 督办中                    │
└───────────────────────────────────────────────┘
```

**跟踪状态视觉：**

| 状态 | 颜色 | 含义 | 进度条颜色 |
|---|---|---|---|
| ✅ 已解决 | 绿色 | 问题已闭环，记录解决天数 | 绿色 100% |
| 🔄 进行中 | 蓝色 | 有响应，正在推进 | 蓝色按比例 |
| ⏳ 未启动 | 灰色 | 已复制/已发出，尚无反馈 | 灰色低比例 |
| 🔴 升级 | 红色 | 问题恶化或范围扩大 | 红色 |

## 三、AI 感知设计体系（v3 新增）

### 3.1 为什么需要 AI 感知

常规数据管理可以通过程序化处理完成（第一层指标卡、右侧履职进度条）。但基于数据定位问题、生成管理建议，这是 AI 的能力。**需要让管理者直观地感受到哪些板块是 AI 在驱动。**

### 3.2 AI 区块标识体系

| 区块 | 身份 | 视觉标识 |
|---|---|---|
| **Layer 1** 今日概览（5 个指标卡） | 程序化数据 | 普通面板，无 AI 标识 |
| **Layer 2 左** 异常诊断 | **AI 扫描中** | `ai-zone-panel` + "AI 扫描中" 脉冲 + "AI 从多维数据中交叉定位异常" |
| **Layer 2 右** 责任主体履职状态 | 程序化数据 | 普通面板，无 AI 标识 |
| **Layer 3** 今日工作提示 | **AI 生成** | `ai-zone-panel` + "✦ AI 生成" + 复制全部按钮 |
| **Layer 4** 异常跟踪 | **AI 持续跟踪** | 面包屑提示 + "AI 持续跟踪异常处置进展，每日自动汇总更新" |

### 3.3 AI 视觉语言

**`.ai-zone-panel`** — AI 驱动面板的专属样式：
- 边框从纯灰变为**掺入强调色的淡蓝调**
- 顶部 2px 的**渐变色上边缘**（从左到右：强调色 → 透明）
- 配合极淡的 `box-shadow`，悬空感略强于数据面板

**`.d-ai-badge` / `.ai-zone-badge`** — AI 标识：
- "AI 扫描中"旁有脉冲圆点动画（`ai-pulse` keyframe）
- "AI 生成"旁有 sparkles 图标

**`.a-insight` 升级** — 每条 AI 分析文字卡：
- 背景从灰色 → **掺 6% 强调色的白色**（极淡蓝底）
- 2px **强调色左边框线**
- 视觉上像是"AI 输出的结论"，而非普通描述文字

**分析标签 `ai-label`** — 每条 AI 分析的类型标记（`关联分析` `交叉验证` `特征分析` `风险推演`）

**目标状态：** 管理者一眼就能区分"哪些是系统自己算的，哪些是 AI 在分析和生成的"。

## 四、场景分类

不同场景的"诊断逻辑"和"行动模型"完全不同，不应混在一个问卷里问。

### 场景 A：日常监管简报（最高优先级）

| 属性 | 定义 |
|---|---|
| **触发条件** | 每天早上的常态 |
| **第一层内容** | 检查任务完成率、隐患新增/整改/超期 |
| **诊断逻辑** | 对比计划进度 + 历史趋势 + 关联分析 |
| **行动特点** | 轻量提醒，可复制的管理文案 |
| **时间粒度** | 按天 |
| **当前原型覆盖** | ✅ v2 已覆盖，v3 升级 AI 深度 |

### 场景 B：重大隐患告警（次优先级）

| 属性 | 定义 |
|---|---|
| **触发条件** | 触发式，紧急 |
| **第一层内容** | 隐患详情、等级、影响范围、当前状态 |
| **诊断逻辑** | 超时未处置 / 未安排复查 / 上报后无响应 |
| **行动特点** | 明确责任人 + 时限 + 升级督办 |
| **时间粒度** | 实时/按小时 |
| **当前原型覆盖** | ❌ 未覆盖 |

### 场景 C：趋势性风险（第三优先级）

| 属性 | 定义 |
|---|---|
| **触发条件** | 周/月维度，战略型 |
| **第一层内容** | 某类隐患反弹、某片区风险上升、某类企业集中出问题 |
| **诊断逻辑** | 趋势分析 + 同类聚集度 + 归因推断 |
| **行动特点** | 系统性措施（专项复核/培训/政策调整） |
| **时间粒度** | 按周/按月 |
| **当前原型覆盖** | ❌ 未覆盖 |

## 五、异常诊断维度体系（完整版）

这是第二层"问题诊断"背后的分类体系。每个维度代表一个可独立配置的诊断规则。

```
异常诊断维度
│
├─ A. 按异常对象（谁/什么异常？）
│   ├─ A1 按人/条线        → 人员履职数据
│   ├─ A2 按片区           → 片区监管数据
│   ├─ A3 按风险等级       → 风险分级数据
│   ├─ A4 按企业           → 企业行为数据
│   └─ A5 按主体类型       → 分类统计数据
│
├─ B. 按异常类型（什么事异常？）
│   ├─ B1 进度异常         → 计划 vs 实际完成
│   ├─ B2 覆盖异常         → 某类主体/片区欠检
│   ├─ B3 质量异常         → 整改闭环慢/超期
│   ├─ B4 行为异常         → 不登录/不自查/不整改
│   └─ B5 趋势异常         → 反弹/上升/聚集
│
├─ C. 按严重程度（多严重？）
│   ├─ C1 高优             → 重大隐患/超期/高危企业
│   ├─ C2 中               → 明显偏离基线
│   └─ C3 低               → 轻微偏差，持续观察
│
├─ D. 基线类型（用什么判断正常？）← 第 0 层
│   ├─ D1 计划对标         → vs 计划/目标值
│   ├─ D2 历史对照         → vs 昨日/上周/上月
│   ├─ D3 同类对比         → vs 其他条线/片区均值
│   └─ D4 阈值告警         → 超时/超限自动触发
│
└─ E. AI 分析类型（AI 做了什么分析？）← v3 新增
    ├─ E1 关联分析         → 发现多维度数据相关性
    ├─ E2 交叉验证         → 多数据源交叉确认问题
    ├─ E3 特征分析         → 识别共同特征模式
    └─ E4 风险推演         → 预测未来风险趋势
```

**诊断引擎规则模板（每条异常 = 一条规则）：**

```
RULE = [对象类型] + [指标] + [对照基线] + [比较条件] + [阈值] → [严重等级]

示例：
  对象: 条线(张毅)
  指标: 今日检查完成率
  基线: 计划值
  条件: 低于计划值 > 15%
  等级: 高优
```

## 六、关键设计决策记录

### 决策 1：领域选择 — 日常监管 > 突发指挥

- **背景：** v1 原型按"突发事件应急指挥"设计（事件列表 + 资源面板）
- **变更：** 根据用户提供的材料，业务重心在"安全监管"——检查任务、隐患整改、企业履职
- **影响：** 信息架构从"事件驱动"改为"诊断驱动"，顶部不再展示事件列表
- **日期：** 2025-07-21

### 决策 2：第一眼原则 — 诊断 > 数据展示

- **决策：** 工作台第一屏的视觉重心应该是"哪里不正常"的诊断结果，而不是指标卡数据
- **理由：** 这是"智能工作台"和"好看大屏"的分水岭。管理者打开工作台的第一需求是"有没有出问题"，而不是"看一堆数字"
- **实现：** 异常诊断区块（第二层）放在工作台中部核心位置，指标卡（第一层）作为顶部概览

### 决策 3：文案原则 — 理念不暴露给用户

- **决策：** "事找人"、"数据找人"是设计理念，不作为页面显性文案展示
- **理由：** 用户应该感受到系统主动推送了重要的事，而不是被告诉"这是事找人功能"
- **实现：** 注释保留，页面标题/区块标题去掉理念文案

### 决策 4：工作提示定位 — 管理者输出工具，非系统内待办

- **决策：** 今日工作提示的定位是"可复制的管理文案"，而不是"系统内待办事项"
- **理由：** 管理者的下级已经能看到系统内的数据，不需要上级在系统内再"触发"他们。管理者的真实动作是复制文案发到微信群/工作群
- **实现：** 去掉勾选框、进度跟踪条、处置按钮；加入"复制全部"功能；每条提示的文案可直接转发
- **日期：** 2025-07-21

### 决策 5：诊断 → 工作提示 一一对应

- **决策：** 每一条异常诊断对应一条可复制的工作提示
- **理由：** 诊断只是发现问题，提示是为了让管理者复制出去往下推进。两者是同一个流程的前后段
- **实现：** 4 条异常诊断 → 4 条工作提示，内容逻辑连续

### 决策 6：持续跟踪而非"昨日跟进"

- **决策：** 异常一经诊断即创建持续跟踪任务，每日更新进展，直到确认关闭
- **理由：** 很多问题的跟进不是一天两天的——需要看到"已跟踪 X 天"、累计进展、最新更新
- **实现：** 跟踪卡显示跟踪天数、进度条、今日最新进展，每天累加，按状态分类
- **日期：** 2025-07-21

### 决策 7：AI 感知可视化

- **决策：** 需要让管理者直观感受到哪些板块是 AI 在驱动
- **理由：** 常规数据程序化处理，AI 定位问题、生成建议——这种能力差异需要被"看见"
- **实现：** AI 驱动区块用 `ai-zone-panel` 样式（渐变色上边+浅蓝调边框+悬空阴影）、AI 标识 badge、分析类型标签、脉冲动画
- **日期：** 2025-07-21

### 决策 8：图标全部使用开源图标库

- **决策：** 全部图标使用 Lucide（MIT 许可证开源图标库），不手绘 SVG
- **理由：** 用户要求使用开源图标库，减少定制维护成本，保持一致的设计语言
- **实现：** 29 个手绘 SVG 全部替换为 `<i data-lucide="...">` 标签 + CDN 运行时渲染
- **日期：** 2025-07-21

### 决策 9：先做场景 A（日常监管简报），后做 B/C

- **优先级：** 日常监管简报 > 重大隐患告警 > 趋势性风险
- **理由：** 日常场景使用频率最高、需求明确、验证成本低
- **当前状态：** 场景 A 的 v3 原型已完成

### 决策 10：三维交叉定位是诊断引擎的核心架构

- **决策：** 异常诊断按"对象 × 类型 × 严重程度"三轴组织
- **理由：** 单一维度（只按类型或只按对象）输出不够精准；三维交叉定位让每条异常可解释、可追溯
- **后续：** 每条异常配置对应一条规则（RULE），规则可独立开关和调参

## 七、v4 方向 — 目标驱动的动态工作台

> **核心转变：** v3 是固定页面布局 + AI 视觉标签；v4 是 AI 作为工作台编排器，根据目标动态组合模块。

### 7.1 产品方向定义

**不是 AI 实时生成任意界面，而是 AI 基于目标、角色、数据上下文，把已有产品模块动态编排成一个工作台。**

核心理念：

> 底层有一套稳定的业务模块和组件，AI 根据当前目标、角色、关注主题、数据状态和权限，把最相关的模块组合出来，并决定展示顺序、下钻路径和推荐动作。

更准确的一句话：

> **底层是稳定的产品模块和业务能力，上层是 AI 根据目标动态组合出来的角色化工作台。**

### 7.2 三层架构

```
┌──────────────────────────────────────────────────────────────┐
│  第三层：AI 编排层                                            │
│  根据目标·角色·数据·偏好 → 决定展示哪些模块、排序、下钻路径      │
├──────────────────────────────────────────────────────────────┤
│  第二层：工作台模块库（可组合积木）                              │
│  指标卡·趋势卡·事项卡·下钻表·履职卡·进度卡·推荐动作卡·汇报卡     │
├──────────────────────────────────────────────────────────────┤
│  第一层：底层业务能力（能力池）                                  │
│  检查任务·隐患整改·自查自纠·专项行动·专家复核·企业画像·风险分级   │
└──────────────────────────────────────────────────────────────┘
```

### 7.3 六对象模型

| 对象 | 定义 | 示例 |
|---|---|---|
| **目标** | 用户当前想推进什么 | 危化专项、重大隐患闭环、九小场所整改 |
| **上下文** | 目标的范围和条件 | 本周、某片区、某类主体、某条责任线 |
| **模块** | 可组合的页面积木 | 指标卡、事项卡、下钻表、行动卡 |
| **工作项** | 底层业务对象 | 任务、异常、跟进事项、核实任务 |
| **行动** | 用户可以推动的动作 | 督办、分派、复核、通知、会议 |
| **规则/偏好** | 决定排序、下钻、提醒和展示 | 重大隐患优先、关注主题相关优先 |

### 7.4 场景配置 → 模块编排

**核心机制：** 每个目标/场景对应一套模块配置。用户切换目标 → 工作台自动重组。

**场景 A：综合监管（默认）**

```
模块：指标卡 → 异常诊断 → 责任主体履职 → 工作提示 → 跟踪任务
下钻：条线 → 片区 → 企业 → 隐患
```

**场景 B：专项检查（如危化专项）**

```
模块：专项进度卡 → 逾期任务卡 → 重大隐患卡 → 专家复核卡 → 推荐动作卡
下钻：片区 → 责任线 → 风险等级 → 企业名单
优先级：重大隐患 > 逾期整改 > 进度落后 > 专家任务超期
```

**场景 C：隐患闭环**

```
模块：重大隐患进度 → 临期整改卡 → 复核任务卡 → 督办建议卡
下钻：隐患等级 → 责任企业 → 整改状态 → 复核结果
```

**场景 D：履职监督**

```
模块：人员履职总览 → 任务完成率 → 异常行为卡 → 督办建议卡
下钻：人员 → 条线 → 任务类型 → 企业
```

### 7.5 AI 编排器的角色

AI 不直接控制 UI 细节。它做的是：

1. **选择模块** — 从模块库中选出与当前目标最相关的模块
2. **排序模块** — 按优先级排列（重大隐患 > 逾期整改 > 进度落后）
3. **配置模块** — 每个模块展示哪些指标、默认下钻到哪个维度
4. **解释模块** — 告诉用户"为什么展示这些"
5. **推荐动作** — 基于当前数据推荐下一步操作

### 7.6 受控动态边界

| 边界 | 规则 |
|---|---|
| 模块受控 | AI 只能从已有模块库中选择 |
| 数据受控 | 每个模块只能读授权数据源 |
| 动作受控 | AI 只能推荐已定义动作（不能直接处罚） |
| 权限受控 | 不同角色看到不同模块和动作 |
| 结论受控 | 区分已确认事实 / 系统推断 / 待核实 |
| 审计受控 | 目标设置、推荐内容、用户采纳全部留痕 |

### 7.7 用户控制

- **调整目标** — 顶部目标栏，可切换预设场景（已实现）；自定义目标输入（待实现）
- **调整模块** — 增删排序模块，偏好自动保存
- **调整下钻** — 改变默认下钻维度
- **偏好记忆** — 下次自动按该偏好组织工作台

### 7.8 Demo 下一版表现

1. 默认展示"综合监管工作台"，四层完整
2. 顶部目标栏显示"当前目标：综合监管"
3. 切换到"危化专项" → 页面动态变化为专项检查模块组合
4. 用户说"把专家执行情况加进来" → 新增模块
5. 用户说"放到前三" → 模块排序调整
6. 系统显示"已记住偏好"

## 十、技术设计决策

### 10.1 图标方案

- **方案：** Lucide 开源图标库（MIT 许可证）
- **加载方式：** CDN `<script src="https://unpkg.com/lucide@latest/dist/umd/lucide.min.js">`
- **渲染方式：** `<i data-lucide="icon-name">` 标签 + `lucide.createIcons()` 运行时渲染为 SVG
- **CSS：** `.lucide { display: inline-block; vertical-align: middle; }`
- **数量：** 共使用 30+ 种不同 Lucide 图标

### 10.2 视觉系统

- **色域：** oklch（Modern minimal 方向：蓝灰色系，强调色 `oklch(56% 0.17 255)`）
- **语义色：** green/orange/red/blue 四色系统
- **字体：** 系统字体栈（SF Pro → system-ui），无自 Web 字体
- **间距：** 8px 基准网格（4/8/16/24/32）
- **圆角：** 6px/8px/12px

### 10.3 设计打磨清单（v3 中完成）

- **可访问性：** `:focus-visible` 聚焦样式、`aria-label`/`aria-hidden`、`.sr-only`
- **交互状态：** hover/disabled/focus-visible/prefers-reduced-motion
- **响应式：** 1100px（3 列指标卡 + 诊断单列）、920px（侧边栏文字简化）、700px（隐藏侧边栏 + 2 列指标卡）
- **空状态：** `.empty-state` 组件（图标 + 标题 + 描述 + 居中布局）
- **溢出保护：** `text-overflow: ellipsis`、`min-width: 0`

## 十一、用户访谈问题建议

以下是针对场景 A（日常监管）设计的精简访谈问题，每条有明确的"信息用途"说明。

### 10.1 关于"正常"的定义（第 0 层基线）

```
Q1: 您说的"进度异常"通常用什么标准判断？
    a. 低于计划/目标值
    b. 低于其他条线平均水平
    c. 相比昨天/上周出现下滑趋势
    d. 凭经验感觉"不太对"
    用途：确定诊断引擎的基线逻辑——是跑对比还是跑阈值。
```

```
Q2: 您每天打开工作台，最先看哪几个数字？
    用途：确定指标卡的优先级和数量上限（5±2）。
```

### 10.2 关于异常的排查路径

```
Q3: 当您发现一个异常（比如某片区检查不足），
    你通常按什么顺序往下排查？
    a. 先看责任人 → 再看具体企业 → 再看隐患清单
    b. 先看风险等级 → 再按片区筛选 → 再看企业
    c. 先看是哪类主体（危化/工矿/建筑）→ 再看具体
    用途：确定钻取路径的默认顺序。
```

```
Q4: 您希望问题定位到什么粒度就够了？
    a. 定位到条线负责人
    b. 定位到具体片区
    c. 定位到具体企业
    d. 定位到具体隐患
    用途：决定工作台的信息层级深度。
```

### 10.3 关于行动闭环

```
Q5: 您发现异常后，一般通过什么方式推进整改？
    a. 系统内下发任务（下级可看到）
    b. 复制文案发微信群/工作群
    c. 电话或口头沟通
    d. 开会时统一布置
    用途：验证"工作提示→可复制文案"的定位是否正确。
```

```
Q6: 您发出的处置指令，最希望系统怎么跟踪？
    a. 告诉我"已下发"就够了
    b. 告诉我"负责人已确认"
    c. 告诉我"预计完成时间 + 实际进展"
    d. 自动跟踪并在超时未完成时再次提醒
    用途：决定第四层的跟踪粒度。
```

### 10.4 关于场景确认

```
Q7: 下面三个场景中，哪个您每天都会遇到？
    哪个您最希望先解决？
    a. 日常检查任务的进度跟踪
    b. 重大隐患的告警和处置
    c. 隐患类型/片区风险的周期性趋势分析
    用途：验证场景优先级排序。
```

## 十二、术语表

| 术语 | 定义 | 备注 |
|---|---|---|
| 第 0 层（预期基线） | 判断"正常"的参照标准 | 计划值/历史均值/同类对比/阈值 |
| 第一层（数据反馈） | 展示发生了什么，附带基线对照 | 程序化处理，无 AI |
| 第二层（问题诊断） | 指出哪里不正常，含 AI 分析深度 | AI 驱动，核心差异化 |
| 第三层（工作提示） | 可复制的管理文案，非系统内待办 | AI 生成，一一对应诊断 |
| 第四层（持续跟踪） | 异常跟踪任务，每日更新，直到关闭 | AI 持续更新 |
| 三维交叉定位 | 异常 = 对象 × 类型 × 严重程度 | 诊断引擎核心架构 |
| AI 分析类型 | 关联分析 / 交叉验证 / 特征分析 / 风险推演 | v3 新增 |
| 基线 | 判断"正常"的参照标准 | 第 0 层 |
| 场景 A | 日常监管简报 | 最高优先级 |
| 场景 B | 重大隐患告警 | 次优先级 |
| 场景 C | 趋势性风险 | 第三优先级 |
| 事找人 | 系统基于角色和场景主动推送需要关注的事 | 设计理念，不暴露给用户 |
| 数据找人 | 系统基于上下文主动展示相关数据 | 设计理念，不暴露给用户 |
| AI 感知 | 让管理者直观感受到哪些板块是 AI 在驱动 | 视觉标识体系 |
| Lucide | MIT 许可证开源图标库 | 全部图标由此库提供 |
| Modern minimal | 当前视觉方向（蓝灰色系 + 单强调色） | SaaS 产品调性 |
| 模块 | 可组合的页面积木，如指标卡、事项卡、下钻表 | v4 核心概念 |
| 目标驱动 | 用户告诉系统当前关注什么，系统动态重组工作台 | v4 核心理念 |
| AI 编排 | AI 根据目标选择模块、排序、配置下钻路径 | AI 的核心角色 |
| 受控动态 | AI 只能从模块库中选择，不能自由生成界面 | v4 安全边界 |
| 场景配置 | 预定义的目标→模块映射关系 | v4 核心数据结构 |
| 模块调整 | 用户可增删排序模块，偏好自动记忆 | v4 用户控制 |
| 六对象模型 | 目标·上下文·模块·工作项·行动·规则/偏好 | v4 核心对象 |
